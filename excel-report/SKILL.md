# Excel 分析报表的呈现风格

> 只定义"导出给非技术受众看的分析 Excel"该长什么样。
> 不管技术实现——用 xlsxwriter 还是 openpyxl、SQL 怎么写、数据怎么拉、怎么连库，那是执行者按当前项目惯例决定的事，本 skill 一律不规定。

## 何时触发

受众是**不写代码的人**（老板、运营、业务方、客户），且以 **Excel 交付**。两个条件同时成立就用。典型信号：

- "给 X 看 / 给老板 / 给业务方 / 给客户 / 发给区域经理"
- "要直观 / 能看懂 / 通俗 / 白话 / 别光给数字 / 弄清楚点 / 像样点"
- "做个报表 / 分析报告 / 数据汇报"
- "拉出来看看趋势 / 对比一下"

用户通常**不会**主动说"画折线图""分多个 Sheet""加口径说明"——他们认为这是理所当然的。所以即使用户只说了"导个 Excel 给老板看"，也按本 skill 的默认呈现来做。

## 不该触发的场景（边界）

- **纯数据导出**：原始字段给下游程序 / BI 工具 / 数据中台消费，没有"给人读"的诉求。
- **单点修改**：在已有 Excel 里加一列、改个格式、写个公式。
- **临时调试**：随手 to_excel 存一下看数据结构。
- **只画图不交付 Excel**：用户只要一张图看趋势（那归图表/可视化能力，不是 Excel 报表）。
- **技术实现选型**：用哪个库、SQL 怎么写——看项目 CLAUDE.md，不归本 skill。

## 核心原则（一切风格的出发点）

**读这份 Excel 的人不写代码、不熟悉字段名、不会自己算结论。**

所以默认行为（除非用户明确说"不用这么麻烦 / 简单点 / 就要原始数据"）：

1. **结论前置**——别让人翻到最后一页才看到答案。
2. **分多个 Sheet**——一个主题一个 Sheet，别把所有东西塞进一张表。
3. **能画图就画图**——趋势用折线、关系用散点、构成/排名用柱状，图比表直观。
4. **白话到底**——口径说明用人话写，不出现裸字段名和 SQL。

这四条是用户每次都要重复嘱咐的，本 skill 把它们变成默认。

## Sheet 结构：总 → 分 → 细 → 口径

按"读者关心度递减"的漏斗组织 Sheet：

1. **总览 / 结论 Sheet（放第一个）**——打开第一眼就看到答案。形式任选其一：
   - 几个 KPI 大卡（核心数字 + 一句话说明，并排）
   - 一段带颜色高亮的结论文字（verdict）
   - 一张最关键的趋势图
2. **分维度下钻 Sheet（中间 1–N 个）**——按业务维度切：分门店 / 分时段 / 分类型 / 分城市 / 分品类……每张表一个切面，别混。
3. **明细 Sheet（按需，放中间偏后）**——原始记录，供核查。
4. **口径说明 Sheet（放最后一个，必备）**——解释"这些数怎么算的、能不能信"。

> 为什么固定这个顺序：非技术读者通常只看第一个 Sheet；想深究的往后翻；质疑数字的人最后看口径兜底。把最枯燥的明细藏中间，把最关键的口径放最后压轴。

## 每张 Sheet 的首屏

- **第 1 行**：大标题，格式 `范围 · 主题`（如"泰国 · 骑手供应趋势"）。字号大、粗体、用项目主色。
- **第 2 行**：副标题，灰色斜体小字，一行交代三件事：`分析周期 | 数据源 | 口径去哪看`。
- **隐藏网格线**——干净的首屏比满屏网格更像"成品"。

> 为什么：打开就知道"这是什么数据、什么时间窗口、口径去哪查"，读者不用回头问。

## 图表：默认配，别只给表

- **时间序列**（按天/周/月趋势）→ 折线图。多指标对比时画在同一张图共享 x 轴。
- **两个变量的关系**（如降水 vs 单量）→ 散点图 + 趋势线，标题里带上相关系数。
- **构成或排名**（各部分占比 / TopN 对比）→ 柱状图或条形图。
- **图标题用白话中文**——写"每日在岗 / 接单骑手数"，不写"online_riders over time"。
- **图放在数据表右侧或下方**，不遮数据；同张 Sheet 图和表都要有——图给结论，表给证据。

> 为什么：老板看图、运营看表。同一份交付里两种读者都要照顾到。

## 白话说明列

关键指标旁边配一句人话解读（斜体灰字），别让人对着列名猜。例：

- "沉默流失：前半段在线、后半段消失（没正式离职也不上线）"
- "损失订单 = 基线应成交 − 实际成交（仅计正值）"

## 口径说明 Sheet（重头戏，区别于"数据导出"的关键）

结构按这个来：

1. **【一句话】这个分析在干什么**——一两行讲清楚目的。
2. **指标定义**——每个指标白话解释怎么算（"总单 ÷ 接单骑手 = 人均单"，不是 SQL）。
3. **为什么这么算（最值钱的一段）**——解释方法论选择，主动回答最可能被质疑的问题。例：
   - "为什么不用全天平均做基线"→ 管控总在高峰开启，全天平均会低估损失。
   - "为什么不用今年 vs 去年同期"→ 控制不住天气变量。
4. **怎么读这些数**——这个数升 / 降分别意味着什么。
5. **局限与偏差（诚实声明）**——估算在哪偏高 / 偏低、哪些东西没剥离、样本量够不够。有偏差的地方用警示色标出。

> 为什么把"为什么这么算"当重点：业务方最爱质疑数字。提前把选择讲清楚，比事后解释省十倍力气。诚实声明局限反而提升可信度——遮遮掩掩才让人不信。

## 颜色与视觉（遵循项目主色，不写死色值）

- 表头、主系列、品牌色 → 用**当前项目的主色**（看项目 CLAUDE.md，本 skill 不规定具体色值）。
- 语义色保持稳定：损失 / 警告用红、达标 / 正向用绿、对比系列用一个主色 + 一个对比色。
- **结论好不好，颜色先说话**——负面结论整段红底，正面绿底，中间态橙。
- 数值格式：大数千分位、百分比留一位小数、金额带币种。
- 重点列用数据条让大小一目了然。

## 与其他规范的分工（互补不重复）

- **颜色美学 / 无障碍 / 明暗适配** → 若当前环境有 `dataviz` skill，参考它。
- **表头样式 / 冻结首行首列 / 筛选 / 文件命名 / debug 后缀 / 分批导出** → 项目 CLAUDE.md。
- **本 skill** → 只管叙事结构、Sheet 划分、图表选型、白话口径。

## 示例

**例 1**
> 用户："把上个月各门店销售拉个 Excel 给区域经理看下，弄清楚点别光甩数字。"
→ Sheet 结构：①总览（3 个 KPI 卡：总 GMV / 同比 / 门店数 + 一张门店排名柱图）②门店明细（按 GMV 降序，重点列加数据条）③口径说明（GMV 含不含退款、为什么排除某门店、统计口径）。

**例 2**
> 用户："骑手这周少了好多，做个分析给运营，要能看出是不是天气原因。"
→ Sheet 结构：①结论（verdict：与天气无关，结构性流失；带颜色高亮）②降水 vs 骑手散点图（标题带相关系数）③按天明细 ④口径（为什么用散点对照而不是今年 vs 去年同期）。

**例 3（不该用本 skill）**
> 用户："把这批订单按 status 筛完导个 csv 给中台入库。"
→ 纯数据导出，下游程序消费。不触发。
